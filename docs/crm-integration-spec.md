# iReparo PBX ↔ CRM Integration Spec

> **Version:** 1.0 · **Date:** 2026-04-12
>
> This document is the contract between the **iReparo PBX** (Go,
> `Centralita-whatomate`) and the **iReparo CRM** (Laravel,
> `reparaciones`). Both sides MUST follow this spec exactly so the
> integration works without manual coordination between the two
> codebases.

---

## 1. Overview

| Direction | Endpoint | Purpose |
|-----------|----------|---------|
| PBX → CRM | `GET  /api/pbx/lookup?phone=<normalized>` | Caller identification on incoming call |
| PBX → CRM | `POST /api/pbx/call-event` | Call lifecycle events (ringing, answered, ended, missed) |

Both endpoints live on the CRM at `https://sat.ireparo.es`.

---

## 2. Authentication

### 2.1 API Key (both endpoints)

Every request from the PBX includes:

```
X-iReparo-Api-Key: <shared_api_key>
```

The CRM verifies with `hash_equals()`. If the key does not match,
respond **403**.

### 2.2 HMAC Signature (POST only)

POST requests additionally include:

```
X-iReparo-Signature:  sha256=<hex_digest>
X-iReparo-Timestamp:  <unix_epoch_seconds>
X-iReparo-Event:      <event_type>
```

**Signed message** (literal dot separator, no JSON parsing):

```
<timestamp>.<raw_request_body>
```

**Algorithm:** HMAC-SHA256 with the shared `webhook_secret`.

**Laravel verification snippet:**

```php
$timestamp = $request->header('X-iReparo-Timestamp');
$signature = $request->header('X-iReparo-Signature');
$rawBody   = $request->getContent();

// 1. Clock skew check (5-minute window)
if (abs(time() - (int) $timestamp) > 300) {
    abort(403, 'Timestamp expired');
}

// 2. HMAC verification
$expected = 'sha256=' . hash_hmac('sha256', $timestamp . '.' . $rawBody, config('ireparo_pbx.webhook_secret'));
if (! hash_equals($expected, $signature)) {
    abort(403, 'Invalid signature');
}
```

### 2.3 Shared secrets

Both secrets are generated by `scripts/deploy.sh` on the PBX VPS:

| PBX config (config.toml)            | CRM .env variable              | Length   |
|--------------------------------------|--------------------------------|----------|
| `integrations.crm.api_key`          | `IREPARO_PBX_API_KEY`          | 24 hex bytes |
| `integrations.crm.webhook_secret`   | `IREPARO_PBX_WEBHOOK_SECRET`   | 32 hex bytes |

## 3. Phone normalization

Both sides MUST normalize phones to **E.164 without the leading `+`**
before comparing or storing:

```
+34 873 94 07 02   →  34873940702
+34 (873) 940-702  →  34873940702
0034873940702      →  34873940702
```

**Algorithm:** strip all non-digit characters, then strip leading `00`.

Go (PBX): `crm.NormalizePhone()`
PHP (CRM): `preg_replace('/\D/', '', $phone)` + `ltrim($result, '00')` if starts with `00`.

### Response codes

| Code | Meaning |
|------|---------|
| 200  | OK (found or not found — check `found` field) |
| 403  | Bad API key |
| 500  | Internal error |

### PBX behavior

- Timeout: **1.5 seconds** (configurable via `lookup_timeout_ms`).
- On timeout or error: show "Unknown caller" in agent panel. Never block.
- Cache: 5 min positive, 30 sec negative (configurable).
- The PBX caches `customer.id` as `external_crm_id` on the local
  Contact row so subsequent calls skip the network round-trip.

## 5. POST /api/pbx/call-event

### Request

```
POST /api/pbx/call-event
Content-Type: application/json
X-iReparo-Api-Key: <api_key>
X-iReparo-Signature: sha256=<hex>
X-iReparo-Timestamp: <unix_epoch_seconds>
X-iReparo-Event: call.ringing
```

### Envelope

Every event body shares the same outer structure:

```json
{
  "event": "call.ringing",
  "timestamp": "2026-04-12T10:30:00.000Z",
  "data": { ... }
}
```

### Response

| Code | Meaning |
|------|---------|
| 200  | Event accepted |
| 403  | Bad signature or API key |
| 422  | Malformed payload (logged, not retried) |

**Always return 200 on success** — the PBX has a retry queue with
exponential backoff (30s → 6h, max 10 attempts) and will re-deliver
on non-2xx responses.

### Idempotency

The CRM MUST use `data.call_id` as an idempotency key. If an event
with the same `call_id` + `event` type arrives again, update the
existing `customer_call_logs` row instead of creating a duplicate.

### 6.1 call.ringing

Fired when an incoming call starts ringing (before anyone answers).

```json

  "timestamp": "2026-04-12T10:30:00.000Z",

  }
}
```

## 7. CRM database schema

### 7.1 customer_call_logs

```sql
CREATE TABLE customer_call_logs (
    id             BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    call_id        VARCHAR(255) NOT NULL,
    event_type     VARCHAR(50)  NOT NULL,
    customer_id    BIGINT UNSIGNED NULL,
    direction      VARCHAR(20)  NOT NULL DEFAULT 'incoming',
    caller_phone   VARCHAR(30)  NOT NULL,
    called_phone   VARCHAR(30)  NOT NULL DEFAULT '',
    pbx_contact_id VARCHAR(255) NULL,
    channel        VARCHAR(30)  NOT NULL DEFAULT 'whatsapp',
    status         VARCHAR(30)  NOT NULL DEFAULT 'ringing',
    duration_seconds INT        NOT NULL DEFAULT 0,
    started_at     TIMESTAMP    NULL,
    answered_at    TIMESTAMP    NULL,
    ended_at       TIMESTAMP    NULL,
    disconnected_by VARCHAR(50) NULL,
    agent_email    VARCHAR(255) NULL,
    agent_name     VARCHAR(255) NULL,
    ivr_flow_name  VARCHAR(255) NULL,
    ivr_path       JSON         NULL,
    recording_url  VARCHAR(500) NULL,
    recording_duration_seconds INT NULL,
    missed_reason  VARCHAR(50)  NULL,
    whatsapp_fallback_sent BOOLEAN NOT NULL DEFAULT FALSE,
    raw_payload    JSON         NULL,
    created_at     TIMESTAMP    NULL,
    updated_at     TIMESTAMP    NULL,

    UNIQUE INDEX idx_call_event (call_id, event_type),
    INDEX idx_customer (customer_id),
    INDEX idx_caller_phone (caller_phone),
    INDEX idx_status (status),
    INDEX idx_created (created_at),

    CONSTRAINT fk_call_logs_customer
        FOREIGN KEY (customer_id) REFERENCES customers(id)
        ON DELETE SET NULL
);
```

### 7.2 customers additions

```sql
ALTER TABLE customers
    ADD COLUMN phone_normalized VARCHAR(30) NULL AFTER phone_alt,
    ADD INDEX idx_phone_normalized (phone_normalized);
```

The CRM should back-fill `phone_normalized` for existing rows on
migration:

```php
DB::table('customers')
    ->whereNotNull('phone')
    ->whereNull('phone_normalized')
    ->eachById(function ($row) {
        $normalized = preg_replace('/\D/', '', $row->phone ?? '');
        if (str_starts_with($normalized, '00')) {
            $normalized = substr($normalized, 2);
        }
        DB::table('customers')
            ->where('id', $row->id)
            ->update(['phone_normalized' => $normalized]);
    });
```

## 9. PBX retry behavior

If the CRM returns non-2xx, the PBX:

1. Persists the event in `crm_event_queue` (Postgres).
2. A background worker retries with exponential backoff:
   30s, 1m, 2m, 5m, 15m, 30m, 1h, 2h, 6h cap.
3. After 10 failed attempts the event moves to `dead_letter` status.
4. The signature + timestamp are frozen at build time, so the CRM
   should NOT re-check clock skew on retried events (the PBX
   re-sends the original headers). **However**, the current CRM
   implementation enforces clock skew — this means retried events
   older than 5 minutes will be rejected with 403. This is
   acceptable for now: the PBX will dead-letter them and they can
   be manually replayed from the admin UI (Phase 3.2).

## 10. Security notes

- All traffic over HTTPS (Let's Encrypt on both sides).
- API key protects against unauthorized access.
- HMAC signature protects against body tampering and replay (5-min window).
- The CRM never trusts `external_crm_id` from the PBX blindly —
  it always verifies the customer exists in its own DB.
- Phone normalization is deterministic and locale-independent.
- Raw payloads are stored for audit/debugging but never exposed to
  end users.
